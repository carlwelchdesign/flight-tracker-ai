# M4.1 — Public Decision Intelligence and Exploration

Default owner: Full-stack product engineering, with product design, aviation
weather, accessibility, and trust review.

## Delivery boundary

This sequence begins after the public FT-404 deployment is complete. FT-403
neutral validation runs after FT-413 through FT-416 so it evaluates the finished
recruiter experience. This milestone turns the
public tracker from a strong map demonstration into an explainable
decision-intelligence experience before M5 explores recommendations or LLM
drafting.

Each ticket uses one dedicated feature branch, intentional ticket-scoped
commits, one pull request targeting `main`, passing required checks, and
recorded browser evidence. No ticket may expose protected tenant data, raw
provider payloads, operational controls, or unsupported flight facts.

## FT-413 — Explain why a selected flight needs attention

Status: Complete

Branch: `feat/ft-413-public-attention-explanation`
Implementation commit: `85afab1`
Merge commit: `2ed8268`
Pull request: [#39](https://github.com/carlwelchdesign/flight-tracker-ai/pull/39)
Owner: Full-stack product engineering and product design

Add a compact, read-only explanation to the selected-aircraft panel so a public
viewer can understand the evidence, deterministic reasoning, uncertainty, and
source freshness behind an attention state without signing in.

Dependencies: FT-404

Acceptance checklist:

- [x] A selected deterministic replay aircraft demonstrates at least one
      meaningful attention state on every supported viewport.
- [x] The explanation separates observed or replay facts, deterministic rule
      results, and geometric estimates; it never describes an estimate as a
      filed route, destination, ETA, or provider observation.
- [x] The panel shows a readable score or priority breakdown including the
      evidence that contributed, source times, freshness, and rule version.
- [x] Live aircraft without the required route/hazard evidence show an honest
      unavailable or not-evaluated state instead of an invented assessment.
- [x] Public responses exclude tenant identifiers, protected alert IDs, raw
      provider payloads, dispatcher actions, notes, and protected evidence URLs.
- [x] The explanation is contextual to the selected aircraft and does not
      restore a marketing banner or walkthrough above the tracker.
- [x] Loading, empty, replay, live, stale, unavailable, and partial-evidence
      states are designed and tested.
- [x] Keyboard and screen-reader users can reach the explanation and understand
      the score without relying on color or map geometry alone.
- [x] Component, Rust contract, authorization, and sanitized-public-boundary
      tests pass.
- [x] Desktop and mobile browser verification proves selection changes the
      explanation and that the public page exposes no operational action.
- [x] Branch, commits, PR, CI results, screenshots, hosted promotion, and
      verification evidence are recorded before completion.

Success signal: In the FT-403 follow-up, a neutral reviewer can identify which
demonstration flight needs attention and explain why without facilitator help.

  FT303 replay state, honest live and replay non-evaluation, bounded parsing,

- Local browser verification at the desktop viewport selected FT303 and showed
  its 85/100 critical priority, four score contributions, rule and score
  versions, replay facts, geometric-estimate disclaimer, and source times with
  no sign-in or operational action.
- A 390 by 844 mobile browser check selected the replay using the public control,
  exposed the same score and evidence before the aircraft facts, reported a
  390-pixel document width with no horizontal overflow, and found no application

- PR #39 passed Rust, web, API/PostGIS, and Vercel checks before merge. Render

## FT-414 — Add a replay time machine and aircraft telemetry charts

Status: Complete

Branch: `feat/ft-414-flight-time-machine`
Implementation commit: `2baab01`
Merge commit: `4fa15e6`
Pull request: [#41](https://github.com/carlwelchdesign/flight-tracker-ai/pull/41)
Owner: Frontend and full-stack product engineering

Let a viewer play, pause, scrub, and replay a bounded demonstration timeline
while the selected aircraft's map position and accessible altitude, speed, and
heading history remain synchronized.

Dependencies: FT-413

Acceptance checklist:

- [x] Replay offers play, pause, restart, scrub, and a small allowlist of speed
      controls with explicit UTC scenario time.
- [x] Map position, observed/replay trail, attention explanation, and telemetry
      charts update from one deterministic virtual clock.
- [x] Altitude, ground speed, and heading charts include units, timestamps,
      focus states, accessible names, and a textual data summary.
- [x] Replay controls are clearly distinguished from live traffic; live ADS-B
      history remains bounded to page memory under ADR-012 and is never
      persisted, exported, logged, or sent to an LLM.
- [x] Scrubbing is deterministic and does not create duplicate alerts, reorder
      evidence incorrectly, or mutate protected operational state.
- [x] Reduced-motion mode replaces nonessential animation while preserving
      time-machine controls and understandable state changes.
- [x] Loading, no-history, single-point, paused, complete, and replay-reset
      states are covered.
- [x] Component and integration tests prove clock synchronization, chart
      calculations, keyboard operation, reduced motion, and reload reset.
- [x] Desktop and mobile browser verification covers scrubbing, playback,
      aircraft selection, charts, weather layers, and no horizontal overflow.
- [x] Branch, commits, PR, CI results, screenshots, hosted promotion, and
      verification evidence are recorded before completion.

Success signal: A first-time viewer can replay the scenario and identify when
the aircraft's attention state changed without needing instructions.

Verification evidence:

- Rust builds the public timeline from the checked-in replay scenario, caps it
  at 15 minutes and 100 observations, and excludes operator, flight, route,
  hazard, alert, provider-record, raw-payload, dispatcher, and audit fields.
- The typed browser boundary rejects malformed, out-of-order, and unbounded
  timelines. Deterministic interpolation tests prove the clock produces the
  same aircraft picture and trail for the same elapsed time.
- Component and integration tests cover transport controls, the bounded speed
  allowlist, scrubbing across the attention threshold, accessible chart text,
  no-history, single-point, unavailable, live, and replay states.
- Local desktop and 390 by 844 mobile browser checks proved play advances the
  clock, pause holds it, restart returns to zero, completion stops playback,
  scrubbing restores FT303's explanation, the map renders a labeled replay
  trail, three telemetry charts render, and the mobile document has no
  horizontal overflow. The reduced-motion browser retained every control while
  collapsing marker transition time.
- PR #41 passed Rust, web, API/PostGIS, and Vercel checks before merge. Render
  staging deploy `dep-d9g8glbrjlhs73bu3hfg` and production deploy
  `dep-d9g8imjrjlhs73bu753g` run merge commit `4fa15e6`; both reported live and
  passed health and readiness checks.
- Vercel deployment `dpl_8X2CfSY87uCx9VeyZgRfk3PNAMiR` was assigned to
  `flight-tracker-ai-one.vercel.app`. Its hosted public proxy returned the
  three-minute scenario, seven bounded observations, and the 0.5x, 1x, and 2x
  playback allowlist with no protected field categories.
- Hosted browser verification opened Replay demo at 16:01 UTC, rendered the
  replay trail and three accessible telemetry charts, advanced playback from

- The production sanitized-public-boundary verifier passed the public tracker,

Status: In progress

Final commit: Pending

Make an aircraft or curated scenario easy to find, revisit, and share while
keeping public query parameters bounded and resilient to expiring live data.

Dependencies: FT-414

Acceptance checklist:

- [x] Search filters only the current sanitized regional picture by displayed

Turn airport markers into inspectable forecast and nearby pilot-report entry
points using bounded, source-attributed NOAA Aviation Weather Center data.

Dependencies: FT-415

Acceptance checklist:

- [ ] Rust owns fixed NOAA Aviation Weather Center TAF and PIREP requests for
      the curated regions or allowlisted airports; the browser cannot submit
      arbitrary coordinates, station lists, time ranges, or upstream URLs.
- [ ] Upstream cadence, result count, timeout, response size, and in-memory
      caching respect the official API guidance and prevent request stampedes.
- [ ] The sanitized public model exposes only fields required for forecast and
      nearby-report presentation, with source, issue/valid times, receive time,
      and freshness or unavailable state.

      METAR observations from TAF forecast periods and conditional changes.
- [ ] Nearby PIREPs show report time, distance, altitude, report type, and
      available turbulence, icing, cloud, wind, or temperature evidence without
      implying complete coverage or current conditions at an aircraft.
- [ ] PIREPs are never silently attributed to a selected flight, and missing or
      ambiguous location/altitude data is handled explicitly.
- [ ] TAF and PIREP failures remain independent from aircraft, METAR, SIGMET,
      radar, satellite, and model-wind layers; the last accepted bounded picture
      is labeled retained rather than current.
- [ ] Provider source links, timestamps, advisory portfolio wording, and
      non-operational limits remain visible.
- [ ] Parser fixtures and Rust/web tests cover malformed, empty, rate-limited,
      stale, retained, and partial upstream responses plus unit/time handling.
- [ ] Desktop and mobile browser verification covers airport selection, a TAF

- [ ] Branch, commits, PR, CI results, screenshots, hosted promotion, and
