# M4.1 — Public Decision Intelligence and Exploration

Default owner: Full-stack product engineering, with product design, aviation
weather, accessibility, and trust review.

## Delivery boundary

This sequence begins after the public FT-404 deployment is complete. FT-403
neutral validation runs after FT-413 through FT-416 so it evaluates the finished
recruiter experience. This milestone turns the
public tracker from a strong map demonstration into an explainable
decision-intelligence experience before M5 explores recommendations or LLM
drafting.

Each ticket uses one dedicated feature branch, intentional ticket-scoped
commits, one pull request targeting `main`, passing required checks, and
recorded browser evidence. No ticket may expose protected tenant data, raw
provider payloads, operational controls, or unsupported flight facts.

## FT-413 — Explain why a selected flight needs attention

Status: Not started

Branch: `feat/ft-413-public-attention-explanation`
Final commit: Pending
Pull request: Pending
Owner: Full-stack product engineering and product design

Add a compact, read-only explanation to the selected-aircraft panel so a public
viewer can understand the evidence, deterministic reasoning, uncertainty, and
source freshness behind an attention state without signing in.

Dependencies: FT-404

Acceptance checklist:

- [ ] A selected deterministic replay aircraft demonstrates at least one
      meaningful attention state on every supported viewport.
- [ ] The explanation separates observed or replay facts, deterministic rule
      results, and geometric estimates; it never describes an estimate as a
      filed route, destination, ETA, or provider observation.
- [ ] The panel shows a readable score or priority breakdown including the
      evidence that contributed, source times, freshness, and rule version.
- [ ] Live aircraft without the required route/hazard evidence show an honest
      unavailable or not-evaluated state instead of an invented assessment.
- [ ] Public responses exclude tenant identifiers, protected alert IDs, raw
      provider payloads, dispatcher actions, notes, and protected evidence URLs.
- [ ] The explanation is contextual to the selected aircraft and does not
      restore a marketing banner or walkthrough above the tracker.
- [ ] Loading, empty, replay, live, stale, unavailable, and partial-evidence
      states are designed and tested.
- [ ] Keyboard and screen-reader users can reach the explanation and understand
      the score without relying on color or map geometry alone.
- [ ] Component, Rust contract, authorization, and sanitized-public-boundary
      tests pass.
- [ ] Desktop and mobile browser verification proves selection changes the
      explanation and that the public page exposes no operational action.
- [ ] Branch, commits, PR, CI results, screenshots, hosted promotion, and
      verification evidence are recorded before completion.

Success signal: In the FT-403 follow-up, a neutral reviewer can identify which
demonstration flight needs attention and explain why without facilitator help.

Status: Not started

Final commit: Pending
Pull request: Pending
Owner: Frontend and full-stack product engineering

Let a viewer play, pause, scrub, and replay a bounded demonstration timeline
while the selected aircraft's map position and accessible altitude, speed, and
heading history remain synchronized.

Dependencies: FT-413

Acceptance checklist:

- [ ] Replay offers play, pause, restart, scrub, and a small allowlist of speed
      controls with explicit UTC scenario time.
- [ ] Map position, observed/replay trail, attention explanation, and telemetry
      charts update from one deterministic virtual clock.
- [ ] Altitude, ground speed, and heading charts include units, timestamps,
      focus states, accessible names, and a textual data summary.
- [ ] Replay controls are clearly distinguished from live traffic; live ADS-B

      evidence incorrectly, or mutate protected operational state.

      time-machine controls and understandable state changes.

      states are covered.

      calculations, keyboard operation, reduced motion, and reload reset.
- [ ] Desktop and mobile browser verification covers scrubbing, playback,
      aircraft selection, charts, weather layers, and no horizontal overflow.
- [ ] Branch, commits, PR, CI results, screenshots, hosted promotion, and
      verification evidence are recorded before completion.

Success signal: A first-time viewer can replay the scenario and identify when

Final commit: Pending
Pull request: Pending

Make an aircraft or curated scenario easy to find, revisit, and share while
keeping public query parameters bounded and resilient to expiring live data.

Dependencies: FT-414

- [ ] Search filters only the current sanitized regional picture by displayed

## FT-416 — Add airport TAF and nearby PIREP intelligence

Status: Not started

Branch: `feat/ft-416-airport-taf-pirep-intelligence`
Final commit: Pending
Pull request: Pending
Owner: Backend, frontend, and aviation-weather engineering

Turn airport markers into inspectable forecast and nearby pilot-report entry
points using bounded, source-attributed NOAA Aviation Weather Center data.

Dependencies: FT-415

Acceptance checklist:

- [ ] Rust owns fixed NOAA Aviation Weather Center TAF and PIREP requests for
      the curated regions or allowlisted airports; the browser cannot submit
      arbitrary coordinates, station lists, time ranges, or upstream URLs.
- [ ] Upstream cadence, result count, timeout, response size, and in-memory
      caching respect the official API guidance and prevent request stampedes.
- [ ] The sanitized public model exposes only fields required for forecast and
      nearby-report presentation, with source, issue/valid times, receive time,
      and freshness or unavailable state.
- [ ] Selecting an airport opens a forecast timeline that distinguishes current
      METAR observations from TAF forecast periods and conditional changes.
- [ ] Nearby PIREPs show report time, distance, altitude, report type, and
      available turbulence, icing, cloud, wind, or temperature evidence without
      implying complete coverage or current conditions at an aircraft.
- [ ] PIREPs are never silently attributed to a selected flight, and missing or
      ambiguous location/altitude data is handled explicitly.
- [ ] TAF and PIREP failures remain independent from aircraft, METAR, SIGMET,
      radar, satellite, and model-wind layers; the last accepted bounded picture
      is labeled retained rather than current.
- [ ] Provider source links, timestamps, advisory portfolio wording, and
      non-operational limits remain visible.
- [ ] Parser fixtures and Rust/web tests cover malformed, empty, rate-limited,
      stale, retained, and partial upstream responses plus unit/time handling.
- [ ] Desktop and mobile browser verification covers airport selection, a TAF
      timeline, nearby PIREPs when available, unavailable fallback, and weather
      layer coexistence.
- [ ] Branch, commits, PR, CI results, screenshots, hosted promotion, and
      verification evidence are recorded before completion.

Success signal: A reviewer can select a regional airport and explain the
difference between its current observation, terminal forecast, and nearby
pilot-reported conditions.

Verification evidence: Pending.
